2-Axis Stabilizer

As a follow up to the previous ball stabilizer project, I wanted to add an additional axis of movement. This project uses 3 MG966R servo motors, and ESP32, and a custom made PCB.

Photo

The frame was designed on SolidWorks and followed the classic 3-RPS Parallel Manipulator mechanics. Final servo angles (a1, a2, a3) were defined via IK (Inverse Kinematics). To find the turn angle for the MG966R servos, the IK followed this pipeline: - Find the tilt angles of the platform (which were the outputs from the x and y PID functions). - Apply the tilt angles to the formula for a plane: Z(x,y) = z0 + ax + by. Where a = arctan(tilt_x) and b = arctan(tilt_y). - Given the height z for a leg, solve for the mechanical angle between the leg and the horizontal - using trig and Law of Cosines. - Convert the mechanical angle to a servo angle using an offset.

Circuit Schematic - KiCad

The purpose behind this PCB is to reduce the need for jumper cables and to make the design more modular. Also, I wanted to gain experience in the whole process of manufacturing a custom PCB: from designing to uploading the Gerber files to a PCB manufacturer.

From the schematic above, the servos are all powered from an external source rather than directly from the ESP32 pins. This is because each ESP32 pin can supply a maximum of 20mA, while each servo draws between 170–290mA under load — far exceeding what the microcontroller can safely provide. For the external power supply, I used a DC power supply. I also conected decoupling capacitors between power and GND to discharge rapid changes in current and background noise.

Through making this board on KiCad, I learned how to assign Net Labels to pin outputs so that there is no need for wire connections and how to assign footprints to each component of the circuit.

PCB Schematic - KiCad

Originally, I traced the power nets on the board, but this was problematic: the traces were too wide because the servos drew large amounts of current. Using DigiKey's PCB Trace Width Conversion Calculator, the traces ended up being 3.17mm wide. The fix: use power and ground planes instead of traces. The ground plane takes up all of the back Copper layer, while the power plane takes up around half of the front Copper layer.

The coding was arguably the most difficult part of this project. Figuring out how Python should communicate with the Arduino via USB, and how to implement a hand tracking system, were both significant challenges. I used MediaPipe, or Google's hand tracking library, to track finger movements in real time. Once finger orientations were identified, I implemented a function to calculate the bend angle of each finger using the dot product formula to find the angle between two vectors: one pointing from the middle joint to the fingertip, and the other from the middle joint to the knuckle. These angles were then sent via USB to the Arduino script.

The hardest challenge came during debugging after the code was complete. Once I ran the Python script, the SG90 servos would track my fingers on the first iteration and then completely stop. The issue was that Python was sending messages faster than the Arduino could process them. By the time the first message was read and sent to the servo motors, there were already 3–4 new messages waiting in the USB buffer. The fix was to discard all but the most recent message on the Arduino side, ensuring the servos always respond to the latest finger position rather than freezing while trying to catch up with a backlog of stale commands.
