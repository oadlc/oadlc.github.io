Robotic Hand

Built a robotic hand that mimics free-floating hand movements. Used Google's Hand tracking library, 5 SG90 servos, and an ATmega328P microcontroller.

Photo
Circuit Schematic

From the schematic above, the servos are all powered from an external source rather than directly from the Arduino pins. This is because each ATmega328P output pin can supply a maximum of 40mA, while each servo draws between 170–290mA under load — far exceeding what the microcontroller can safely provide.

For the external supply, I used two 18650 lithium-ion batteries. I initially planned to wire them in series for a combined 7.4V output and use a 5V buck converter to step the voltage down. However, due to the current charge levels of the batteries — one measuring 4V and the other 1.3V — the combined voltage was approximately 5.3V, which is within the acceptable operating range for the SG90 servos. This made the buck converter unnecessary, simplifying the circuit.

The coding was arguably the most difficult. Particularly, trying to figure how Python should communicate to the Arduino via USB, and how to implement a hand tracking system was challenging. For this project, I implemented Mediadpipe - Google's Hand Tracking library to be able to track finger movements. Once finger orientations were deciphered, I implemented a function to find the bend angle of each finger. To do this, I used the dot product formula to find cos(theta) between two vectors. One vector pointed from the middle joint to the tip of the finger, and the other vector pointed from the middle joint to the kunckle of the finger. These angles were then sent via USB to the uploaded Arduino script.

The hardest challenge was debugging upon completion of the code. Once I ran the Python script, the SG90 servos would track my fingers for the first iteration, and then completely stop after. The issue lied in the fact that Python was sending messages too quickly for the Arduino. By the time that the very first Python message was procecced by the Arduino script, and then sent to the servo motors, there would be around 3-4 new Python messages waiting in the USB buffer. The fix: instead of trying to read all of the Python messages, read only the most recent one. This was mostly handled on the Arduino script side of the code, where all of the Python messages are discarded but the last one. This ensures that the servos do not freeze while trying to keep up with all of the messages in the buffer.
